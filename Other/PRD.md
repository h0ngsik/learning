# PRD란?
PRD는 **Product Requirements Document**의 약자로 제품 요구사항 정의서를 말한다.
> ⭐️ 왜 제품을 만들어야하고, 누구를 위한 것이며, 무엇을 만들 건지를 합의하는 문서다.
## '어떻게'가 아닌 '무엇'을 정의하는 문서
PRD는 **무엇을 왜 만드는지를 정의**하는 것이지, 어떻게 구현하는지를 정의하지 않는다.
> ⭐️ 기술 스택, 아키텍처, DB 설계 같은건 PRD가 아니라 기술 설계 문서의 영역이다.
## PRD의 목적
PRD의 본질은, 모든 이해관계자가 같은 제품을 만들고 있다는 것을 확인하는 것이다.
> PM·디자이너·엔지니어 등 모든 이해 관계자들이 같은 제품을 만들고 있다는 것을 확인하는 합의서다.

제품을 만들다 보면,
> 🙋‍♂️ "이거 원래 하기로 한게 맞나요?", "이건 왜 구현하는거에요?"

하는 순간들을 마주하게 된다. 그 때 같이 확인할 수 있는 기준이 PRD다.

그래서 분량이 아니라 합의하는 밀도가 중요하다고 생각한다.
<br>
한 페이지든, 열 페이지든 모든 이해 관계자가 같은 제품을 상상하고 있다면 좋은 PRD일 것 같다.

# PRD를 작성하기 위한 질문
PRD는 결국 다섯 개의 질문에 답하는 문서다.<br>
1. 왜 만드는가?
2. 누구를 위한건가?
3. 무엇을 만드는가?
4. 성공의 기준은 무엇인가?
5. 무엇을 안 할 건가?

이 질문들에 대한 답변이 또렷하다면 어떤 형식이든 PRD의 역할은 할 수 있을거라고 생각한다.

## 1️⃣ 배경과 문제 정의 (Why)
어떤 사용자가, 어떤 상황에서, 무엇 때문에 불편한지를 작성한다.
여기서 흔한 실수가 문제를 적어야하는 곳에 **이미 정해둔 해결책을 적는 것**이다.

예를 들어,

> ❌ "우리 가게에는 키오스크가 없어요!"

이게 문제일까?
<br><br>
키오스크가 없는 건 문제가 아니다. 사실 키오스크를 도입하자라는 **해결책을 거꾸로 문제처럼 말**하고 있는 것 뿐이다.
<br><br>
그럼 어떤게 진짜 문제일까?
> ✅ "점심에 손님이 몰리면 주문 줄이 길어져서 손님이 그냥 나가버려요!"

이렇게 진짜 문제가 무엇인지 정의할 수 있어야 한다.

### 진짜 문제를 적으면, 해결책이 여러개가 보인다
줄이 길어서 손님이 나가는게 문제라면, 
- 키오스크도 답이될 수 있지만 직원을 한 두명 더 두거나,
- 미리 주문을 받을 수 있는 앱을 활용하는 등

여러 방법을 도입할 수 있다.

그런데 처음부터 "키오스크가 없는게 문제야!" 라고 말한다면, 다른 해결책을 고민하기가 어려워진다.

### 이걸 구분하는 방법
#### 1. 문장의 주어를 보자.
주어가 기능이면 위장된 문제, 사람이면 진짜 문제다.
#### 2. "문제 안에 이미 '어떻게'가 들어갔나?"를 물어보자.
해결책을 적은 또 다른 문제 예시
> ❌ "수강생용 디스코드 채널이 없어요."

-> "디스코드 채널을 추가하자!"라는 해결책을 이미 제시하고 있다.

불편한 상황으로 다시 정의한 예시
> ✅ "수강생이 막혔을 때 질문할 곳이 없어서, 혼자 끙끙대다 포기해요."

-> 정말 불편한 상황만 존재한다.
#### 3. "왜?"를 한 번 더 물어보자.
> "키오스크가 없어요!" -> "왜 그게 문제야?" -> **"주문이 느려서 손님이 나가요!"**

한 번만 더 물어봐도 진짜 문제를 파악할 수 있다.

### 좋은 문제 정의
좋은 문제 정의는 데이터나 사용자의 목소리로 뒷받침된다.

"그럴 것 같아요"보다는,
> "이탈률 70%, 인터뷰 8명 중 6명이 같은 불편을 겪고 있어요."

처럼 근거가 뒷받침되어야 한다.

## 2️⃣ 사용자 정의 (Who)
이 기능을 사용할 핵심 사용자가 누구인지 정의한다.

많은 사람들이 “모든 사람”을 대상으로 제품을 만들고 싶어 하지만, 실제로는 **특정 문제를 가장 크게 느끼는 사용자**를 먼저 정의해야 한다.

예를 들어 다음 사용자는 모두 학습 서비스의 사용자일 수 있다.

- 온라인 강의를 수강하는 직장인
- 개발 공부를 시작한 대학생
- 이직을 준비하는 주니어 개발자

하지만 각 사용자는 서로 다른 상황과 문제를 가지고 있다.

따라서 PRD에서는 단순히 사용자의 직업이나 나이를 적는 것이 아니라, 다음 질문에 답해야 한다.
> “누가 어떤 문제를 겪고 있는가?”

### 좋은 사용자 정의
좋은 사용자 정의는 단순히 사람을 설명하는 것이 아니라, 그 사람이 처한 상황과 문제를 함께 설명한다.

다음과 같은 내용을 함께 포함해보자.
- 어떤 사람인가?
- 어떤 상황에 있는가?
- 어떤 문제를 겪고 있는가?
- 무엇을 기대하는가?

예를 들어, 다음과 같이 작성하는 것이 훨씬 구체적이다.

> 개발 공부를 시작한 대학생
>
> 프로그래밍 경험은 적지만 취업을 위해 개발을 배우고 있다.<br>
> 수많은 강의 중 무엇을 먼저 들어야 할지 몰라 학습 방향을 잡지 못하고 있다.

이렇게 작성하면 제품이 해결해야 할 문제가 자연스럽게 드러난다.

### 여러 사용자가 존재한다면?
제품에는 여러 사용자가 존재할 수 있다.

예를 들어, LMS라면 다음과 같은 사용자 그룹이 존재한다.
- 수강생
- 강사
- 관리자

하지만 모든 사용자가 동일한 중요도를 가지는 것은 아니다.

수강생을 중심으로 설계되는 LMS도 있고,
강사의 운영 편의를 중심으로 설계되는 LMS도 있다.

따라서 PRD에서는
> “이 제품이 가장 먼저 해결하려는 사용자는 누구인가?”

를 명확하게 정의하는 것이 중요하다.

핵심 사용자가 명확할수록 제품의 방향성과 우선순위를 결정하기 쉬워진다.

예를 들어,
> 수강생이 원하는 강의를 쉽게 찾지 못하는 것이 문제라면?

강의 탐색 경험이 우선순위가 될 수 있다.

반대로,
> 강사가 수강생을 관리하기 어렵다는 것이 문제라면?

출석 관리나 과제 관리 기능이 우선순위가 될 수 있다.

결국 같은 LMS라도 누구의 문제를 먼저 해결하려고 하는지에 따라 제품의 모습은 달라질 수 있다.

## 3️⃣ 요구사항 (What)
요구사항을 작성하는 형태는 두 가지다.
1. 사용자 스토리(User Story)
2. 기능 요구사항(Functional Requirement)

### 사용자 스토리 (User Story)
사용자 스토리는 제품을 사용하는 사람의 입장에서 “무엇을 하고 싶은지”를 작성한다.

일반적으로 아래와 같은 형식을 사용한다.
> "~로서(As a), ~하기 위해(So that), ~하고 싶다(I want to)"
- **~로서 (역할/사용자)**: 이 기능을 사용하는 핵심 사용자는 누구인가?
- **~하기 위해 (목적/가치)**: 이 행동을 통해 사용자가 얻는 가치는 무엇인가?
- **~하고 싶다 (행동)**: 사용자가 구체적으로 하려는 행동은 무엇인가?

예를 들어,
> 학습자로서, 원하는 주제를 빠르게 찾기 위해, 강의를 카테고리로 필터링하고 싶다.

이걸 자연스럽게 풀어쓰면 다음과 같다.
> 학습자가 원하는 주제를 빠르게 찾기 위해 강의를 카테고리로 필터링할 수 있다.

사용자 스토리는 다음과 같이 구성된다.
- **역할**: 학습자
- **목적**: 원하는 주제를 빠르게 찾기 위해
- **행동**: 강의를 카테고리로 필터링하기

### 기능 요구사항 (Functional Requirement)
기능 요구사항은 사용자 스토리를 만족시키기 위해 시스템이 제공해야하는 기능을 구체적으로 작성한다.

사용자 스토리가 "왜 필요한가?"를 설명한다면, 기능 요구사항은 "무엇을 만들어야 하는가?"를 설명한다.

예를 들어,
아래와 같은 사용자 스토리가 있다고 가정해보자.
> 학습자로서, 원하는 주제를 빠르게 찾기 위해, 강의를 카테고리로 필터링하고 싶다.

이 사용자 스토리를 만족시키기 위한 기능 요구사항은 다음과 같이 작성할 수 있다.
- 사용자는 강의 카테고리를 선택할 수 있어야 한다.
- 선택한 카테고리에 해당하는 강의만 표시되어야 한다.
- 사용자는 선택한 카테고리를 해제할 수 있어야 한다.
- 사용자는 필터를 초기화할 수 있어야 한다.

기능 요구사항은 가능한 한 **구체적이고 명확하게 작성**해야 한다.

좋은 기능 요구사항은 구현 여부를 쉽게 판단할 수 있고, 개발자와 디자이너가 동일하게 이해할 수 있다.

### 우선순위
모든 요구사항을 한 번에 개발할 수는 없다.

시간과 자원은 제한되어 있기 때문에 가장 중요한 기능부터 개발해야 한다.

우선순위를 정하는 방법은 여러 가지가 있지만, 대표적으로 P0/P1/P2 방식과 MoSCoW 방식을 많이 사용한다.

#### P0 / P1 / P2
기능이나 이슈의 중요도를 기준으로 우선순위를 나누는 방식이다.

일반적으로 다음과 같이 구분한다.
- P0 (필수)
- P1 (중요)
- P2 (선택)

예를 들어 온라인 강의 서비스라면 다음과 같이 구분할 수 있다.

| 기능 요구사항 | 우선순위 |
|--------------|----------|
| 회원가입 및 로그인 | P0 |
| 강의 목록 조회 | P0 |
| 강의 카테고리 필터 | P1 |
| 강의 좋아요 기능 | P2 |
| 강의 공유 기능 | P2 |

#### MoSCoW
MoSCoW는 이번 릴리스에 어떤 기능을 포함할지 결정할 때 사용하는 방식이다.

일반적으로 다음과 같이 구분한다.

- **Must** : 반드시 포함되어야 하는 기능
- **Should** : 가능하면 포함하는 기능
- **Could** : 여유가 되면 포함하는 기능
- **Won’t** : 이번 버전에서는 제외하는 기능

예를 들어 MVP를 개발한다면 다음과 같이 정리할 수 있다.

- **Must** : 회원가입, 로그인, 강의 조회
- **Should** : 카테고리 필터
- **Could** : 좋아요 기능
- **Won**’t : 추천 알고리즘

#### 주의할 점
P0/P1/P2와 MoSCoW는 비슷해 보이지만 같은 개념은 아니다.

P0/P1/P2는 기능이나 이슈의 중요도를 구분하는 방식이고,<br>
MoSCoW는 이번 릴리스에 포함할 개발 범위를 결정하는 방식이다.

따라서 P1이 반드시 Should가 되거나, P2가 반드시 Could가 되는 것은 아니다.

중요도가 낮은 기능이라도 전략적인 이유로 이번 버전에 포함될 수 있고,<br>
중요도가 높은 기능이라도 일정이나 리소스 문제로 다음 버전으로 미뤄질 수 있다.

#### 어떤 방식을 사용해야 할까?
정답은 없다.

팀에 따라 P0/P1/P2만 사용하기도 하고, MoSCoW만 사용하기도 한다.

또는 두 방식을 함께 사용하기도 한다.

중요한 것은 어떤 방식을 사용하느냐보다, 다음과 같이 팀이 명확하게 합의하는 것이다.

> “무엇이 가장 중요한 기능인지”
> 
>“이번 버전에서 무엇을 만들고 무엇을 만들지 않을 것인지”

## 4️⃣ 성공 기준 (Success Metrics)
제품을 만들었다고 해서 성공한 것이 절대 아니다.

사용자의 문제가 실제로 해결되었는지 확인할 수 있어야 한다.

성공 기준은
> "이 기능이 잘 동작하고 있는지 어떻게 판단할 것인가?"

에 대한 답이라고 생각한다.

## 5️⃣ 제외 범위 (Out of Scope)
