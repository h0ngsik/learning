# PRD란?
PRD는 **Product Requirements Document**의 약자로 제품 요구사항 정의서를 말한다.
> ⭐️ 왜 제품을 만들어야하고, 누구를 위한 것이며, 무엇을 만들 건지를 합의하는 문서다.
## '어떻게'가 아닌 '무엇'을 정의하는 문서
PRD는 **무엇을 왜 만드는지를 정의**하는 것이지, 어떻게 구현하는지를 정의하지 않는다.
> ⭐️ 기술 스택, 아키텍처, DB 설계 같은건 PRD가 아니라 기술 설계 문서의 영역이다.
## PRD의 목적
PRD의 본질은, 모든 이해관계자가 같은 제품을 만들고 있다는 것을 확인하는 것이다.
> PM·디자이너·엔지니어 등 모든 이해 관계자들이 같은 제품을 만들고 있다는 것을 확인하는 합의서다.

제품을 만들다 보면,
> 🙋‍♂️ "이거 원래 하기로 한게 맞나요?", "이건 왜 구현하는거에요?"

하는 순간들을 마주하게 된다. 그 때 같이 확인할 수 있는 기준이 PRD다.

그래서 분량이 아니라 합의하는 밀도가 중요하다고 생각한다.
<br>
한 페이지든, 열 페이지든 모든 이해 관계자가 같은 제품을 상상하고 있다면 좋은 PRD일 것 같다.

# PRD를 작성하기 위한 질문
PRD는 결국 다섯 개의 질문에 답하는 문서다.<br>
1. 왜 만드는가?
2. 누구를 위한건가?
3. 무엇을 만드는가?
4. 성공의 기준은 무엇인가?
5. 무엇을 안 할 건가?

이 질문들에 대한 답변이 또렷하다면 어떤 형식이든 PRD의 역할은 할 수 있을거라고 생각한다.

## 1️⃣ 배경과 문제 정의 (Why)
어떤 사용자가, 어떤 상황에서, 무엇 때문에 불편한지를 작성한다.
여기서 흔한 실수가 문제를 적어야하는 곳에 **이미 정해둔 해결책을 적는 것**이다.

예를 들어,

> ❌ "우리 가게에는 키오스크가 없어요!"

이게 문제일까?
<br><br>
키오스크가 없는 건 문제가 아니다. 사실 키오스크를 도입하자라는 **해결책을 거꾸로 문제처럼 말**하고 있는 것 뿐이다.
<br><br>
그럼 어떤게 진짜 문제일까?
> ✅ "점심에 손님이 몰리면 주문 줄이 길어져서 손님이 그냥 나가버려요!"

이렇게 진짜 문제가 무엇인지 정의할 수 있어야 한다.

### 진짜 문제를 적으면, 해결책이 여러개가 보인다
줄이 길어서 손님이 나가는게 문제라면, 
- 키오스크도 답이될 수 있지만 직원을 한 두명 더 두거나,
- 미리 주문을 받을 수 있는 앱을 활용하는 등

여러 방법을 도입할 수 있다.

그런데 처음부터 "키오스크가 없는게 문제야!" 라고 말한다면, 다른 해결책을 고민하기가 어려워진다.

### 이걸 구분하는 방법
#### 1. 문장의 주어를 보자.
주어가 기능이면 위장된 문제, 사람이면 진짜 문제다.
#### 2. "문제 안에 이미 '어떻게'가 들어갔나?"를 물어보자.
**❌ 해결책을 적은 또 다른 문제 예시**
> "수강생용 디스코드 채널이 없어요."

-> "디스코드 채널을 추가하자!"라는 해결책을 이미 제시하고 있다.

**✅ 불편한 상황으로 다시 정의한 예시**
> "수강생이 막혔을 때 질문할 곳이 없어서, 혼자 끙끙대다 포기해요."

-> 정말 불편한 상황만 존재한다.
#### 3. "왜?"를 한 번 더 물어보자.
> "키오스크가 없어요!" -> "왜 그게 문제야?" -> **"주문이 느려서 손님이 나가요!"**

한 번만 더 물어봐도 진짜 문제를 파악할 수 있다.

### 좋은 문제 정의
좋은 문제 정의는 데이터나 사용자의 목소리로 뒷받침된다.

"그럴 것 같아요"보다는,
> "이탈률 70%, 인터뷰 8명 중 6명이 같은 불편을 겪고 있어요."

처럼 근거가 뒷받침되어야 한다.

## 2️⃣ 사용자와 정의 (Who)
이 기능을 사용할 핵심 사용자가 누구인지 정의한다.

예를 들어,
- 온라인 강의를 수강하는 직장인
- 개발 공부를 시작한 대학생
- 이직을 준비하는 주니어 개발자

## 3️⃣ 요구사항 (What)
요구사항을 작성하는 형태는 두 가지다.
1. 사용자 스토리(User Story)
2. 기능 요구사항(Functional Requirement)

### 사용자 스토리 (User Story)
사용자 스토리는 제품을 사용하는 사람의 입장에서 “무엇을 하고 싶은지”를 작성한다.

일반적으로 아래와 같은 형식을 사용한다.
> "~로서(As a), ~하기 위해(So that), ~하고 싶다(I want to)"
- **~로서 (역할/사용자)**: 이 기능을 사용하는 핵심 사용자는 누구인가?
- **~하기 위해 (목적/가치)**: 이 행동을 통해 사용자가 얻는 가치는 무엇인가?
- **~하고 싶다 (행동)**: 사용자가 구체적으로 하려는 행동은 무엇인가?

예를 들어,
> 학습자로서, 원하는 주제를 빠르게 찾기 위해, 강의를 카테고리로 필터링하고 싶다.

이걸 자연스럽게 풀어쓰면 다음과 같다.
> 학습자가 원하는 주제를 빠르게 찾기 위해 강의를 카테고리로 필터링할 수 있다.

사용자 스토리는 다음과 같이 구성된다.
- **역할**: 학습자
- **목적**: 원하는 주제를 빠르게 찾기 위해
- **행동**: 강의를 카테고리로 필터링하기

### 기능 요구사항 (Functional Requirement)
기능 요구사항은 사용자 스토리를 만족시키기 위해 시스템이 제공해야하는 기능을 구체적으로 작성한다.

사용자 스토리가 "왜 필요한가?"를 설명한다면, 기능 요구사항은 "무엇을 만들어야 하는가?"를 설명한다.

예를 들어,
아래와 같은 사용자 스토리가 있다고 가정해보자.
> 학습자로서, 원하는 주제를 빠르게 찾기 위해, 강의를 카테고리로 필터링하고 싶다.

이 사용자 스토리를 만족시키기 위한 기능 요구사항은 다음과 같이 작성할 수 있다.
- 사용자는 강의 카테고리를 선택할 수 있어야 한다.
- 선택한 카테고리에 해당하는 강의만 표시되어야 한다.
- 사용자는 선택한 카테고리를 해제할 수 있어야 한다.
- 사용자는 필터를 초기화할 수 있어야 한다.

기능 요구사항은 가능한 한 **구체적이고 명확하게 작성**해야 한다.

좋은 기능 요구사항은 구현 여부를 쉽게 판단할 수 있고, 개발자와 디자이너가 동일하게 이해할 수 있다.

### 우선순위
모든 요구사항을 한 번에 개발할 수는 없다.

시간이 무제한이라면 좋겠지만, 그렇지 않기 때문에 제한된 시간과 리소스로 가장 중요한 기능부터 개발해야한다.

일반적으로는 다음과 같이 구분한다.
- P0 (필수)
- P1 (중요)
- P2 (선택)

예를 들어 온라인 강의 서비스라면 다음과 같이 우선순위를 나눌 수 있다.

| 기능 요구사항 | 우선순위 |
|--------------|----------|
| 회원가입 및 로그인 | P0 |
| 강의 목록 조회 | P0 |
| 강의 카테고리 필터 | P1 |
| 강의 좋아요 기능 | P2 |
| 강의 공유 기능 | P2 |

다만 스타트업이나 MVP 개발에서는 MoSCoW 방식을 사용하기도 한다.

- **Must (반드시)**: 이 기능이 없으면 출시 자체에 의미가 없는 것.
- **Should (가능하면)**: 있으면 좋지만, 없어도 일단 출시할 수 있는 것.
- **Could (여유가 되면)**: 시간이 남으면 넣는 것.
- **Won't (이번엔 제외)**: 이번 버전에서는 명시적으로 하지 않는 것.
